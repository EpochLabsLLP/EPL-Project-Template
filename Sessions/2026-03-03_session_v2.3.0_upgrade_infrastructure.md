## What Was Accomplished

### Design Discussion: Deploying to Projects with Older Versions
- Identified three gaps in the template sync system for upgrade scenarios
- Established that CLAUDE.md does NOT need updating on routine syncs because governance logic lives in rules/hooks (infrastructure, auto-overwritten) — CLAUDE.md works by reference, not by content
- Decided against automatic CLAUDE.md replacement; migration should be guided and human-in-the-loop
- Chose bottom-up build order: marker → merge → skill

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| CLAUDE.md is reference-based, not content-based | Governance logic in rules/hooks (auto-updated); CLAUDE.md rarely needs changing |
| settings.json uses managed_scaffolding category | Permissions block is project-owned; hooks block is template-managed |
| Hook identity = command string | `$CLAUDE_PROJECT_DIR/.claude/hooks/{name}.sh` is unique per hook |
| claude_md_version is independent from template_version | Infrastructure can update (v2.3→v2.5) without CLAUDE.md structure changing |
| /template-migrate is human-in-the-loop | Touching a project's constitution deserves human review |
